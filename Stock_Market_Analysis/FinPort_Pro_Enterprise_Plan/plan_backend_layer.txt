 1. ARCHITECTURE OVERVIEW
================================================================================

The backend consists of 10 independent microservices, each deployed as
a separate Spring Boot application with its own repository, database
schema, and independent release cycle.

Inter-service communication:
  - Synchronous:  REST APIs (via API Gateway)
  - Asynchronous: Apache Kafka (event-driven)
  - ZERO direct database queries between services

Design Principles:
  - Single Responsibility per service
  - Own your data (each service owns its schema)
  - Event-driven for real-time updates
  - Stateless services (horizontal scaling)
  - Circuit breaker for fault isolation
  - Distributed tracing for observability

 4.1 API GATEWAY (finport-gateway)
--------------------------------------------------------------------

Tech: Spring Cloud Gateway

Key Responsibilities:
  - Central entry point for all client requests
  - JWT validation (signature check, no DB call)
  - Route management to downstream services
  - Rate limiting (token bucket — Redis backed)
  - Circuit breaker (Resilience4j)
  - CORS enforcement
  - Request logging (correlation ID injection)

Route Configuration:
  /api/auth/**           → finport-user-service:8081
  /api/portfolios/**     → finport-portfolio-service:8082
  /api/prices/**         → finport-market-service:8083
  /api/risk/**           → finport-risk-engine:8084
  /api/analytics/**      → finport-analytics-service:8086
  /api/alerts/**         → finport-notification-svc:8087
  /api/ai/**             → finport-ai-service:8088
  /ws/**                 → WebSocket upstream

Filters:
  JwtValidationFilter    → pre-filter, validates JWT, injects user context
  RateLimiterFilter      → 100 req/min per IP
  CircuitBreakerFilter   → opens on 5 consecutive failures, 30s half-open
  LoggingFilter          → adds X-Correlation-ID header, logs req/res
  CorsFilter             → allows configured origins

Rate Limiting:
  Algorithm: Token Bucket
  Storage:   Redis (per-IP bucket)
  Limits:    100 req/min (normal), 1000 req/min (admin)

 4.2 USER SERVICE (finport-user-service)
--------------------------------------------------------------------

Endpoints:
  POST /api/auth/register
    → Body: { email, password, fullName }
    → Response: { userId, email }

  POST /api/auth/login
    → Body: { email, password }
    → Response: { accessToken, refreshToken, user }

  POST /api/auth/refresh
    → Cookie: refreshToken
    → Response: { accessToken }

  POST /api/auth/logout
    → Blacklist current token in Redis

  GET /api/users/me
    → Returns authenticated user profile

  PUT /api/users/roles/{userId}
    → Admin only: change user role

JWT Strategy:
  Access Token:  RS256, 15-minute expiry, no DB lookup needed
  Refresh Token: 7-day expiry, stored in HttpOnly secure cookie
  Token Blacklist: Redis set with TTL matching token expiry

Password Hashing: BCrypt (12 rounds)

Event Published:
  user-registered → topic: user-events

 4.3 PORTFOLIO SERVICE (finport-portfolio-service)
--------------------------------------------------------------------

Endpoints:
  GET    /api/portfolios                     → List user's plans
  POST   /api/portfolios                     → Create new plan
  GET    /api/portfolios/{id}                → Get plan details
  PUT    /api/portfolios/{id}                → Update plan
  DELETE /api/portfolios/{id}                → Soft delete plan

  GET    /api/portfolios/{id}/holdings       → List all holdings
  POST   /api/portfolios/{id}/holdings       → Add holding
  PUT    /api/portfolios/{id}/holdings/{hid} → Update holding
  DELETE /api/portfolios/{id}/holdings/{hid} → Remove holding

Business Logic:
  - On Add Holding:
    → If Target 1/2/3 blank: auto-fill (+6% / +10% / +15%)
    → If Stop Loss blank: auto-fill (-5%)
    → If Risk Level blank: default "Medium"
    → If Strategic Role blank: default "Tactical Swing"
    → If Volatility blank: default "Medium"
    → Allocation Rs = Entry Price * Quantity

  - On any holding change:
    → Publish: portfolio-updated event to Kafka

Validation (Jakarta Validation):
  @NotBlank: symbol, sector
  @Positive: entryPrice, quantity
  @PastOrPresent: entryDate

Event Published:
  portfolio-updated → Kafka topic: portfolio-updated
    Key: portfolioId
    Payload: { portfolioId, userId, holdings[], timestamp }

Caching:
  Redis: portfolio:{userId}:list → 10 min TTL
  Invalidated on: portfolio-updated event

 4.4 MARKET DATA SERVICE (finport-market-service)
--------------------------------------------------------------------

Endpoints:
  GET /api/prices/{symbol}/latest
    → Returns: { symbol, price, changePct, volume, refreshedAt }

  GET /api/prices/{symbol}/history?from=&to=
    → Returns: [ { date, open, high, low, close, volume } ]

    → Returns: [ { symbol, price, changePct } ]

Event Consumer:
  price-ingested → from Scraper Service
    → Upsert latest_prices table
    → Insert into price_history (TimescaleDB)
    → Publish: price-updated (enriched event)

Event Published:
  price-updated → Kafka topic: price-updated
    Key: symbol
    Payload: { symbol, price, changePct, volume, source, timestamp }

  Redis: price:latest:{symbol} → 60 sec TTL
  Redis: price:batch → 60 sec TTL (batch of all symbols)

TimescaleDB:
  price_history table is a hypertable partitioned by month
  Retention: 3 years of daily data
  Archival: Monthly archive to S3 Parquet files

 4.5 RISK ENGINE SERVICE (finport-risk-engine)
--------------------------------------------------------------------

This is the CORE calculation service — ZERO endpoint exposure.
Pure event-driven.

Event Consumer:
  price-updated → recalculate risk for all holdings with that symbol
  portfolio-updated → recalculate risk for all holdings in plan

Calculation Engine:
  Investment Value     = Entry Price × Quantity
  Current Value        = Current Price × Quantity
  P&L ₹               = Current Value − Investment Value
  P&L %               = (Current Price − Entry) / Entry
  Risk Amount          = (Entry − Stop Loss) × Quantity
  SL Exit Value        = Stop Loss × Quantity
  Dist to SL %         = (Current − SL) / Current
  Dist to T1 %         = (T1 − Current) / Current
  Reward:Risk Ratio    = (T1 − Entry) / (Entry − SL)
  Expected Value       = RR × Risk Amount
  Action               = IF cur <= SL → "Exit"
                         IF cur >= T1 → "Book Profit"
                         ELSE "Hold"

Post-Calculation:
  → Upsert risk_schema.holding_risk_metrics
  → Check: IF current_price <= stop_loss
      → Publish: stoploss-triggered
  → Check: IF current_price >= target_1
      → Publish: target-hit
  → Always publish: risk-recalculated

Events Published:
  risk-recalculated → topic: risk-recalculated
    Key: portfolioId
  stoploss-triggered → topic: stoploss-triggered
    Key: holdingId
  target-hit → topic: target-hit
    Key: holdingId

Consistency:
  Eventual consistency model
  Risk metrics may lag 5-10 seconds behind price change
  Acceptable for tactical portfolio (not HFT)

Idempotency:
  Risk calculation is idempotent — same price yields same metrics
  No double-publish: Redis lock per holdingId (5 sec TTL)

 4.6 SCRAPER SERVICE (finport-scraper-service)
--------------------------------------------------------------------

This is a STATELESS service — no database.
All output goes to Kafka.

Scraper Sources:
  1. NSE/BSE Official API    → Primary source (most reliable)
  2. MoneyControl             → Web scraping (Selenium/Playwright)
  3. Groww API                → REST API (public endpoints)
  4. Economic Times           → RSS Feed parsing
  5. Mint                     → RSS Feed parsing
  6. CNBC                     → API (market data endpoints)
  7. Way2News                 → API for news sentiment

Scheduling (Spring Scheduler + Quartz):
  Market Hours (9:15 AM – 3:30 PM IST):
    - Price scraping: every 5 minutes
  Pre/Post Market:
    - Price check: every 30 minutes
  EOD (6:00 PM IST):
    - OHLCV data storage run
  News Scraping:
    - Every 60 minutes (all day)

Pipeline per Scrape Cycle:
  1. Scrape raw data from source
  2. Validate: price > 0, symbol exists, timestamp recent
  3. Deduplicate: Redis key (symbol + timestamp hash), 10 min TTL
  4. Transform: normalize to standard PriceEvent schema
  5. Publish to Kafka: price-ingested topic

Event Published:
  price-ingested → topic: price-ingested
    Key: symbol
    Payload: { symbol, price, open, high, low, close,
               volume, source, scrapedAt }

Error Handling:
  - If source fails → retry 3x with exponential backoff
  - If all retries fail → log error + publish to scrape-failed DLT
  - Use last known price (no stale publication)
  - Alert ops team if a source fails 3 consecutive cycles

Anti-Detection (Scraping):
  - Rotating residential proxy pool
  - Random User-Agent headers
  - Request throttling (1 req/sec per source)
  - Headless browser with realistic viewport

 4.8 NOTIFICATION SERVICE (finport-notification-svc)
--------------------------------------------------------------------

Pure event consumer — NO REST endpoints (except health check).

Event Consumer:
  stoploss-triggered → Send critical alert (email + push + in-app)
  target-hit         → Send positive alert (email + push + in-app)
  risk-threshold     → Send warning (email only)

Notification Channels:
  1. WebSocket Push → real-time in-app notification
  2. Email → AWS SES / SendGrid
  3. Push (Future) → Firebase Cloud Messaging (mobile)

WebSocket Server:
  - Maintains active connections per userId
  - On relevant event → push JSON to client socket
  - Connection authenticated via JWT

Alert Payload:
  {
    "alertId": "uuid",
    "type": "STOPLOSS_TRIGGERED",
    "severity": "CRITICAL",
    "title": "Stop Loss Hit — TRIDENT",
    "body": "TRIDENT dropped to ₹24.90, below SL ₹25.24",
    "symbol": "TRIDENT",
    "timestamp": "2026-03-03T10:30:00Z",
    "read": false
  }

Deduplication:
  Redis: alert:sent:{holdingId}:{type} → 1 hour TTL
  Prevents duplicate alerts for same SL/Target event

--------------------------------------------------------------------

Architecture: Spring Boot wrapper + internal Python FastAPI service

Event Consumer:
  portfolio-snapshot-requested (or direct REST call)

    → Body: { portfolioId, riskProfile, macroContext }
    → Response: 202 Accepted (async — result via Kafka)

  GET /api/ai/analysis/{portfolioId}/latest

AI Engine (Python FastAPI):
  - Receives enriched portfolio data
  - Stock Classification: Defensive / Cyclical /
                          Commodity Beneficiary / High Beta
  - Sentiment Scoring: Positive / Neutral / Negative
  - Directional Bias: Bullish / Bearish / Neutral
  - Scenario Generation: Bull / Bear / Neutral plans
  - Action Matrix: Add / Hold / Reduce / Exit per stock
  - LLM Integration: OpenAI / Gemini API for reasoning

Event Published:
  ai-analysis-completed → topic: ai-analysis-completed
    Key: portfolioId
    Payload: { portfolioSummary, stockAnalysis[], scenarios,
               actionMatrix[], timestamp }

  ai_schema.analysis_snapshots → full JSON + timestamp

Producer → Consumer Mapping:
--------------------------------------------------------------------
Topic                  Producer         Consumer(s)
--------------------------------------------------------------------
price-ingested         Scraper          Market Data Service, Audit
price-updated          Market Data      Risk Engine, Analytics, AI, Audit
portfolio-updated      Portfolio        Risk Engine, Analytics, Audit
risk-recalculated      Risk Engine      Analytics, Notification, AI, Audit

ai-analysis-completed  AI Service       WebSocket (push to UI), Audit
user-events            User Service     Analytics (user count), Audit

Ordering Guarantees:
  - price-updated: ordered per symbol (partition key = symbol)
  - portfolio-updated: ordered per portfolio (key = portfolioId)
  - stoploss-triggered: ordered per holding (key = holdingId)

Idempotency:
  - All events carry unique eventId (UUID)
  - Consumers maintain processed-event set in Redis (24h TTL)
  - On duplicate eventId → skip processing

Retry Policy:
  - 3 retries with exponential backoff (2s → 4s → 8s)
  - After 3 failures → send to Dead Letter Topic (*.DLT)
  - DLT monitored via Grafana alert

Exactly-Once Semantics:
  - Kafka transactions enabled for stoploss-triggered and target-hit
  - Idempotent producer: enable.idempotence=true
  - Consumer: read_committed isolation level

================================================================================
 7. LOOSE COUPLING STRATEGY
================================================================================

How Services Remain Decoupled:
--------------------------------------------------------------------
1. NO cross-service foreign keys
   - Portfolio stores userId as UUID, NOT a FK to user table
   - Risk stores holdingId as UUID, NOT a FK to holding table

2. Communication is event-based
   - Portfolio does NOT call Risk Engine API
   - Portfolio publishes 'portfolio-updated' → Risk Engine consumes

3. Each service owns its data
   - Market Data owns price_history
   - Risk Engine owns risk_metrics
   - Analytics owns denormalized KPI tables

4. Shared data is replicated via events
   - When Market Data receives a new price → publishes event
   - Risk Engine REACTS to event (doesn't poll Market Data)

5. Schema changes are versioned
   - Events carry "version" field
   - Consumers handle multiple versions gracefully
   - Old versions deprecated after 2 release cycles

6. No shared libraries (except common DTOs)
   - Each service has its own Exception handling
   - Each service has its own Validation rules
   - Only share: finport-common (event models ONLY)

  - 1 Deployment + 1 Service per microservice
  - HPA: CPU threshold 70%, min 2, max 10 pods
  - Pod disruption budget: maxUnavailable = 1
  - Health probes: /actuator/health (liveness + readiness)
  - ConfigMaps for non-sensitive config
  - Secrets for DB passwords, JWT keys, API keys
