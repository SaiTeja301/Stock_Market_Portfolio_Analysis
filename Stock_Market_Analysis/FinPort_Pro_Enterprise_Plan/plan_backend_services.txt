================================================================================
 SERVICE 1: API GATEWAY (finport-gateway)
================================================================================

Purpose:
  Central entry point. Routes, authenticates, rate-limits, logs.
  NO business logic. NO database.

Architecture Style: Reactive Gateway (Spring Cloud Gateway - WebFlux)
  WHY: Non-blocking I/O for high-throughput routing. Gateway handles
  thousands of concurrent connections вҖ” blocking threads would bottleneck.

Kafka Produces: NONE
Kafka Consumes: NONE
Database Schema: NONE
Communication: REST (inbound) вҶ’ REST (outbound routing)

Config Strategy:
  - Routes defined in application.yml (declarative)
  - JWT public key loaded from K8s Secret / env var
  - Redis URL from ConfigMap
  - Rate limits configurable per environment

Exception Handling:
  - All errors return JSON: { status, error, path, timestamp }
  - 429 for rate limit, 401 for invalid JWT, 503 for circuit open

Security:
  - Validates JWT signature ONLY (no DB call)
  - Injects X-User-Id, X-User-Role headers for downstream services
  - Strips Authorization header before forwarding (optional)

Testing:
  - WireMock for downstream service stubs
  - TestContainers for Redis
  - Full route integration tests

Scalability:
  - Min 3 pods (always available)
  - HPA on active connections (not CPU)
  - Stateless вҖ” scales horizontally without limit

 SERVICE 3: PORTFOLIO SERVICE (finport-portfolio-service)
================================================================================

  Manages Plans (portfolios) and Holdings. Core CRUD aggregate.
  Publishes portfolio-updated events on every mutation.

Kafka Produces: portfolio-updated
Kafka Consumes: NONE (produces only)
Database Schema: portfolio_schema
Communication: REST (inbound) + Kafka (outbound events)

================================================================================
 SERVICE 4: MARKET DATA SERVICE (finport-market-service)
================================================================================

Purpose:
  Stores live + historical stock prices. Caches latest prices in Redis.
  Converts raw scraper data into enriched price events.

Architecture Style: Hexagonal Architecture (Ports & Adapters)
  WHY: Multiple external integrations (scraper input, TimescaleDB,
  Redis cache, Kafka output). Hexagonal isolates core from adapters.

Kafka Produces: price-updated
Kafka Consumes: price-ingested (from Scraper)
Database Schema: market_schema (TimescaleDB)
Communication: REST (query APIs) + Kafka (in + out)

================================================================================
 SECTION D: DEPLOYMENT SEPARATION STRATEGY
================================================================================

Each service is deployed independently:

Service                 Replicas(min)  Replicas(max)  HPA Trigger
--------------------------------------------------------------------
API Gateway             3              10             Active connections
User Service            2              5              CPU 70%
Portfolio Service       2              8              CPU 70%
Market Data Service     2              6              CPU 70%
Risk Engine             3              15             Kafka consumer lag
Scraper Service         2              4              CPU 70%
Analytics Service       2              6              CPU 70%
Notification Service    2              6              Kafka consumer lag
AI Service              1              4              CPU + queue depth
Audit Service           2              4              Kafka consumer lag
--------------------------------------------------------------------

Deployment Order (dependency chain):
  1. Kafka cluster (prerequisite)
  2. PostgreSQL + Redis (prerequisite)
  3. API Gateway
  4. User Service
  5. Portfolio Service + Market Data Service (parallel)
  6. Risk Engine
  7. Analytics + Notification + AI + Audit (parallel)
  8. Scraper Service (last вҖ” triggers data flow)


================================================================================
 SECTION E: SCALABILITY STRATEGY PER SERVICE
================================================================================

Service               Strategy
--------------------------------------------------------------------
API Gateway           Stateless. Add pods. Nginx ingress load balances.
User Service          Stateless. Redis token cache shared across pods.
Portfolio Service     Stateless. Redis cache invalidated via events.
Market Data Service   Read replicas for history queries. Redis for latest.
Risk Engine           Stateless. Scale by Kafka partitions (12 partitions
                      = up to 12 parallel consumer instances).
Scraper Service       Split symbol list across pods. Each pod scrapes a
                      subset. Coordinated via Redis distributed lock.
Analytics Service     Read replicas. Pre-computed materialized views.
Notification Svc      Stateless. WebSocket sessions sticky via Redis.
AI Service            Queue-based scaling. LLM calls are slow вҖ” scale
                      based on pending analysis queue depth.
Audit Service         Partitioned tables by month. Archival to S3.
--------------------------------------------------------------------


================================================================================
 SECTION F: FUTURE EXTENSIBILITY STRATEGY
================================================================================

Adding new service (e.g., Options Trading):
  1. Create new repo: finport-options-service
  2. New Kafka topic: options-expiry-triggered
  3. Risk Engine: add new consumer for options events
  4. Analytics: add options exposure aggregator
  5. UI: add new lazy module /features/options/
  6. Gateway: add route /api/options/** вҶ’ new service
  7. ZERO changes to Portfolio, Market Data, User services

Adding new scraper source:
  1. Add new class in scraper/source/ (implements SourceAdapter)
  2. Register in SourceRegistryConfig
  3. ZERO changes to pipeline, publisher, or other sources

Adding new notification channel (SMS):
  1. Add SmsChannel.java (implements NotificationChannel)
  2. Register in ChannelRegistry
  3. ZERO changes to consumers or other channels

Adding new AI model:
  1. Add adapter in ai/llm/ (implements LlmClient)
  2. Switch via config: llm.provider=gemini
  3. ZERO changes to orchestrator, parser, or enricher
